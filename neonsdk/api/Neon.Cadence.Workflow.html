    GetVersionAsync(String, Int32, Int32)

    Used to implement backwards compatible changes to a workflow implementation.

    Declaration
    public async Task<int> GetVersionAsync(string changeId, int minSupported, int maxSupported)
    Parameters
    Type Name Description
    String changeId

    Identifies the change.
    Int32 minSupported

    Specifies the minimum supported version. You may pass DefaultVersion (-1) which will be set as the version for workflows that haven't been versioned yet.
    Int32 maxSupported

    Specifies the maximum supported version.
    Returns
    Type Description
    Task<Int32>

    The workflow implementation version.
    Remarks

    It is possible to upgrade workflow implementation with workflows in flight using the GetVersionAsync(String, Int32, Int32) method. The essential requirement is that the new implementation must execute the same logic for the decision steps that have already been executed and recorded to the history fo a previous workflow to maintain workflow determinism. Subsequent unexecuted steps, are free to implement different logic.

    note

    Cadence attempts to detect when replaying workflow performs actions that are different from those recorded as history and will fail the workflow when this occurs.

    Upgraded workflows will use GetVersionAsync(String, Int32, Int32) to indicate where upgraded logic has been inserted into the workflow. You'll pass a changeId string that identifies the change being made. This can be anything you wish as long as it's not empty and is unique for each change made to the workflow. You'll also pass minSupported and maxSupported integers. minSupported specifies the minimum version of the workflow implementation that will be allowed to continue to run. Workflows start out with their version set to DefaultVersion or (-1) and this will often be passed as minSupported such that upgraded workflow implementations will be able to take over newly scheduled workflows.
    maxSupported essentially specifies the current (latest) version of the workflow implementation.

    When GetVersionAsync(String, Int32, Int32) called and is not being replayed from the workflow history, the method will record the changeId and maxSupported values to the workflow history. When this is being replayed, the method will simply return the maxSupported value from the history. Let's go through an example demonstrating how this can be used. Let's say we start out with a simple two step workflow that first calls ActivityA and then calls ActivityB: 

    public class MyWorkflow : WorkflowBase
{
    public async Task DoSomething()
    {
        var activities = Workflow.NewActivityStub<MyActivities>();

        await activities.ActivityAAsync();  
        await activities.ActivityBAsync();  
    }
}

    Now, let's assume that we need to replace the call to ActivityA with a call to ActivityC. If there is no chance of any instances of MyWorkflow still being in flight, you could simply redepoy the recoded workflow: 

    public class MyWorkflow : WorkflowBase
{
    public async Task<byte[]> RunAsync(byte[] args)
    {
        var activities = Workflow.NewActivityStub<MyActivities>();

        await activities.ActivityCAsync();  
        await activities.ActivityBAsync();
    }
}

    But, if instances of this workflow may be in flight you'll need to deploy a backwards compatible workflow implementation that handles workflows that have already executed ActivityA but haven't yet executed ActivityB. You can accomplish this via: 

    public class MyWorkflow : WorkflowBase
{
    public async Task<byte[]> RunAsync(byte[] args)
    {
        var activities = Workflow.NewActivityStub<MyActivities>();
        var version    = await GetVersionAsync("Replace ActivityA", DefaultVersion, 1);    

        switch (version)
        {
            case DefaultVersion:

                await activities.ActivityAAsync();  
                break;

            case 1:

                await activities.ActivityCAsync();  // <-- change
                break;
        }

        await activities.ActivityBAsync();  
    }
}

    This upgraded workflow calls GetVersionAsync(String, Int32, Int32) passing minSupported=DefaultVersion and maxSupported=1 For workflow instances that have already executed ActivityA, GetVersionAsync(String, Int32, Int32) will return DefaultVersion and we'll call ActivityA, which will match what was recorded in the history. For workflows that have not yet executed ActivityA, GetVersionAsync(String, Int32, Int32) will return 1, which we'll use as the indication that we can call ActivityC.

    Now, lets say we need to upgrade the workflow again and change the call for ActivityB to ActivityD, but only for workflows that have also executed ActivityC. This would look something like: 

    public class MyWorkflow : WorkflowBase
{
    public async Task<byte[]> RunAsync(byte[] args)
    {
        var activities = Workflow.NewActivityStub<MyActivities>();
        var version    = await GetVersionAsync("Replace ActivityA", DefaultVersion, 1);    

        switch (version)
        {
            case DefaultVersion:

                await activities.ActivityAAsync();  
                break;

            case 1:

                await activities.ActivityCAsync();  // <-- change
                break;
        }

        version = await GetVersionAsync("Replace ActivityB", 1, 2);    

        switch (version)
        {
            case DefaultVersion:
            case 1:

                await activities.ActivityBAsync();
                break;

            case 2:

                await activities.ActivityDAsync();  // <-- change
                break;
        }
    }
}

    Notice that the second GetVersionAsync(String, Int32, Int32) call passed a different change ID and also that the version range is now 1..2. The version returned will be DefaultVersion or 1 if ActivityA and ActivityB were recorded in the history or 2 if ActivityC was called.

    MutableSideEffectAsync(Type, String, Func<Object>)

    Calls the specified function and then searches the workflow history to see if a value was already recorded with the specified id. If no value has been recorded for the id or the value returned by the function will be recorded, replacing any existing value. If the function value is the same as the history value, then nothing will be recorded.

    This version of the method uses a parameter to specify the expected result type.

    Declaration
    public async Task<object> MutableSideEffectAsync(Type resultType, string id, Func<dynamic> function)
    Parameters
    Type Name Description
    Type resultType

    Specifies the result type.
    String id

    Identifies the value in the workflow history.
    Func<Object> function

    The side effect function.
    Returns
    Type Description
    Task<Object>

    The latest value persisted to the workflow history.
    Remarks

    Here we call NewActivityFutureStub<TActivityInterface>(String, ActivityOptions) specifying "foo" as the workflow method name. This matches the [ActivityMethod(Name = "foo")] decorating the FooAsync() activity interface method. Then we start the first activity by awaiting ActivityFutureStub<TActivityInterface>. This returns an IAsyncFuture<T> whose GetAsync() method returns the activity result. The code above calls this to retrieve the result from the first activity after executing the second activity in parallel.

    note

    You must take care to pass parameters that match the target method. Neon.Cadence does check these at runtime, but there is no compile-time checking for this scheme.

    You'll also need to cast the GetAsync() result to the actual type (if required). This method always returns the object type even if referenced workflow and activity methods return void. GetAsync() will return null in these cases.
