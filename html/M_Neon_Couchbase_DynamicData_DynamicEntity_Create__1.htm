Remarks

This method is capable of instantiating derived entities in a future-proof manner. This is possible for entities whose defining interfaces were tagged with a Type value. This is set to a unique identifier for the type within the application domain.

Entities serialize their own type identifier as well as the identifiers for any inherited types to the internal EntityTypePathName JSON property. The type identifiers are formatted into a list of colon (:) separated values with the current type's identifier listed first, followed by its parent's identifier and so on, all the way to the root entity's identifier.

This identifier list provides the information necessary to dynamically instantiate the derived entity that best fits the data. For example, say you release a product that defines a base IProduct interface with ICandy derived from it. IProduct defines a Name as property and ICandy adds Calories.

An application can persist a base IProduct instance to the database as well as a derived ICandy. Calling this method to load the base IProduct or ICandy document will simply load it. Calling this method to load IProduct but passing the ICandy document, will actually create an ICandy instance but return it cast to the base type (IProduct).

Applications can test entity types using the usual C# is/as operators or switch on the entity's type property, if one was tagged using IsTypeProperty.

This also enables future-proofing. Say version 1.0 of an application only knew about IProduct and ICandy. Then, version 2.0 is released that adds IBeer which also derives from IProduct and persists some beer to a database. Loading an IBeer document as an IProduct from version 1.0 will still work, even though v1 is not aware of this type. An IProduct entity will be returned (what else can be done), but the IBeer properties will still be loaded into the underlying JObject.
